# Sigil Constitution

## Core Principles

### I. Privacy-First

All raw telemetry (file paths, terminal commands, git metadata, process names, window focus) MUST stay on the engineer's machine. Only summarized, user-reviewed context may be sent to external LLMs via cloud inference. Fleet reporting sends only anonymized aggregate counts and rates — never raw events, file paths, code content, or query text. The engineer has a kill switch (`sigilctl purge`) to wipe the local store at any time. Data flows MUST be inspectable via the Insights view and `sigilctl export`. Privacy is the product's trust foundation; violating it is an existential risk.

### II. Daemon-First

The daemon (`sigild`) is the core product and the first thing to validate. The observation → analysis → suggestion loop must be useful on its own, without the shell or OS. New capabilities start in the daemon and prove value before being surfaced through higher layers (shell, OS). The daemon MUST run stable for 48+ hours, under 50MB RSS, on constrained hardware (2017 MacBook Pro). If a feature cannot be validated at the daemon layer, it is not ready for the shell layer.

### III. Interface-Driven Design

Consumers MUST depend on interfaces, not concrete types. Every cross-package boundary uses an explicit Go interface (`store.EventReader`, `inference.Backend`, `collector.Source`, `notifier.Platform`, `actuator.Actuator`). Mocks are generated by mockery from these interfaces. This enables isolated unit testing without real databases, real LLMs, or real OS services. New packages MUST define or consume interfaces at their boundaries — never accept `*ConcreteType` in a public API when an interface will do.

### IV. DAG Discipline

The package dependency graph MUST be a directed acyclic graph. `event` is the leaf package with zero internal imports. No package may import a package that would create a cycle. New packages MUST be placed in the DAG before implementation begins. The dependency order is: `event` → `config` → `store` → `inference` → `collector` → `notifier` → `analyzer` → `actuator` → `fleet` → `socket` → `cmd/sigild`. Violations of the DAG are build-blocking defects.

### V. Progressive Trust

The suggestion system earns the right to act through demonstrated utility. Five discrete notification levels (Silent → Digest → Ambient → Conversational → Autonomous) gate how aggressively the system surfaces suggestions. Confidence scores increase with observation count — the daemon never shows a suggestion until it has observed the pattern multiple times. At Level 4 (Autonomous), only actions with >90% confidence and a reversible undo path may auto-execute. The first week is the noisiest; by week three, only suggestions the engineer acts on survive. Trust is earned, never assumed.

### VI. Minimal Dependencies

The daemon uses pure Go with no CGo requirement. SQLite via `modernc.org/sqlite` (pure Go). No web frameworks, no ORMs, no dependency injection frameworks. Dependencies MUST be justified by concrete need, not convenience. The dependency surface stays tight: `fsnotify`, `go-toml/v2`, `modernc.org/sqlite`. New dependencies require explicit justification. The daemon is a single statically-linked binary with fast startup and low memory footprint.

### VII. Observable by Default

Every subsystem MUST be queryable. The Unix socket server exposes JSON-over-socket methods for all daemon state. `sigilctl` provides CLI access to events, suggestions, patterns, config, fleet preview, and AI queries. All data stored in SQLite is exportable via `sigilctl export`. Structured logging (`slog`) is used throughout. The daemon's behavior MUST be understandable from its outputs alone — no black boxes.

## Technical Constraints

### Hardware Target

The reference platform is a 2017 MacBook Pro (Intel, 8-16GB RAM). If Sigil runs well on constrained hardware, it validates the cost story for enterprise deployment on any fleet hardware. Memory budgets: daemon ~50MB, local inference model via zero-copy mmap, shell ~200MB.

### Concurrency Model

SQLite single-writer: `store.Open` sets `SetMaxOpenConns(1)`. Do not add concurrent writers. Context propagation is mandatory — `context.Background()` is only used at goroutine roots without a parent context. All blocking operations accept and respect `ctx`.

### Protocol Stability

The Unix socket protocol is newline-delimited JSON. One `Request` line, one `Response` line per connection. Adding new methods requires only a `srv.Handle(...)` call — no protocol changes. The protocol shape is a contract with the shell and CLI.

## Development Workflow

### Test Discipline

Tests use table-driven patterns with `t.Run` subtests. Integration tests use real SQLite (in-memory via `openMemory(t)`), not mocks, for the store layer. Mocks are reserved for cross-package boundaries (inference backends, OS notification platforms). Coverage target: >50% enforced by `make coverage`, aspirational >90%. `make check` (fmt + vet + test) MUST pass before every commit.

### Commit Discipline

Each logical change is one commit. Phase work follows dependency order — never skip an issue or batch commits across issues. Build verification (`go build ./... && go vet ./... && go test ./...`) runs after every change. Commits reference issue numbers when closing GitHub issues.

### Code Quality

Idiomatic Go per [Effective Go](https://go.dev/doc/effective_go). No dead code — if it's unused, delete it. No duplication — shared helpers go in the leaf package (`event`) or the consuming package. No over-engineering — three similar lines are better than a premature abstraction. Error handling is explicit; errors are wrapped with context using `fmt.Errorf("...: %w", err)`.

## Governance

This constitution governs all development on the `wambozi/sigil` repository. It supersedes ad-hoc practices and guides AI agent behavior during spec-driven development. Amendments require documentation of rationale, review by the project maintainer, and a version bump below.
